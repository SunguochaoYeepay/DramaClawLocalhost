# Delivery Boundaries

交付边界的核心是：**只交付用户请求对应的业务结果，不交付内部接口痕迹。**

## 业务结果 vs. 技术痕迹

可交付的是：

- 肖像、身份图、草图、视频等业务产物
- 当前任务状态
- 接口明确返回的可用相对路径

不可交付的是：

- API 端点
- JSON 参数
- 调试路径
- 端点确认过程
- 伪造或猜测出来的结果链接

## 文件路径

- 如果接口明确返回 `*_url` 这类相对路径，直接把相对路径当结果交付。
- 不要自行拼接 host、域名或其他前缀。
- 不要把 `/api/v1/...` 当成用户结果链接。
- 二进制文件默认不 fetch；只在接口已经给出结果路径时把路径交给用户。
- 对于肖像、身份图这类生成型图片产物，如果接口已经返回相对路径，就直接交付该相对路径；不要再做 `HEAD`/`GET` 可达性探测。
- 收到 `image_url`、`portrait_url` 或同类业务结果路径后，默认立即结束，不再补做“最小校验”“直连校验”或下载路由猜测。
- 不要把相对路径转换成自造的绝对 URL 再交给用户。
- 不要为了确认 mp4/wav/png/jpg 是否可访问而重复探测同一路径。
- 单 beat 视频重做 POST 成功后不要重复 POST 同一 beat；没有 `result.video_path` 时不要探测 `/files`，不要探测 `/api/v1/projects/{project}/files`，只能按已启动或当前状态收口。
- `generated: true` 不等于返回了新视频路径；没有 `result.video_path` 时，不要交付 beat 记录里的旧 `video_url`，不要拼接 host。

## 最终成片

- 只有用户明确要求链接、下载、打开或给他看成片时，才进入最终成片交付判断。
- 如果只是“继续生成到成片 / 合成完成”，默认含义是按当前 `next_step` 逐轮推进并汇报状态；本轮最多启动一个写任务，不等于索要最终成片链接。
- 如果接口已经返回正式相对路径，默认只交付该相对路径并立即收口；不要再补 host、绝对 URL、“可直接打开”外链或下载直链。
- 接口没有返回正式文件路径时，只能说明“已完成，但当前接口未返回可直接打开的文件路径”。
- 如果 `compose_episode` 已完成，但当前返回里没有正式结果路径，就在这里立即收口：
  - 直接告诉用户“已完成，但当前接口未返回可直接打开的文件路径”
  - 不再继续猜测相对路径
  - 不再继续探测文件路由、导出路由或下载直链
  - 不再重新触发 `compose`
  - 不再为了找路径去查实现细节、OpenAPI、mock 路由或其它内部说明

### 成片交付例外

仅当 `compose_episode` 已明确完成，且用户明确要求拿到可打开结果时，才允许做一次轻量结果确认。

- 只针对最终成片
- 只做精确确认，不做目录猜测和批量探测
- 验证失败就回落到“已完成，但当前接口未返回可直接打开的文件路径”
- “精确确认”只允许基于接口已返回的正式结果字段进行一次确认；如果当前接口根本没有返回结果路径，就不再做额外确认

## 二进制资源

- 不要为确认结果去批量猜测 mp4/png/jpg/mp3/zip 路径。
- 只有接口明确返回结果路径时，才把它交付给用户。
- 没有 `result.video_path` 时不要探测 `/files` 或 `/api/v1/projects/{project}/files`；不要把 beat 记录里的旧 `video_url` 当作这次重做的可交付结果。
- 不要把 `host.docker.internal`、localhost、测试域名或 API host 拼进用户结果。

## 禁止

- 不要把 API 地址伪装成最终交付
- 不要通过大范围路径探测来“找文件”
- 不要在用户没要时主动扩大交付范围
- 不要在“已完成但无正式结果路径”时继续 `HEAD/GET` 文件路径、导出路径或重新触发 `compose`
